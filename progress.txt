## Codebase Patterns

### Architecture
- **Backend**: Python FastAPI with WebSocket relay to LLM providers
- **Frontend**: Next.js 14 with Zustand for state management
- **Communication**: WebSocket for real-time audio streaming and message passing

### LLM Provider Pattern
- All LLM clients must implement the same interface: `connect()`, `send_audio()`, `receive_messages()`, `disconnect()`, `is_connected`
- Use `get_llm_client(provider)` factory function in `backend/services/openai_relay.py` to get the appropriate client
- Provider selection can come from: (1) frontend UI, (2) environment variable `LLM_PROVIDER`, (3) legacy `USE_MOCK_OPENAI` flag
- Message types for provider responses: `conversation.item.input_audio_transcription.completed` (transcription), `response.text.done` (suggestions)

### Gemini-Specific Patterns
- **Model names**: Use short names like `gemini-2.5-flash`, `gemini-2.0-flash` (not `models/gemini-...`)
- **Available models** (as of Jan 2025): `gemini-2.5-flash`, `gemini-2.0-flash`, `gemini-2.0-flash-lite`, `gemini-flash-latest`, `gemini-pro-latest`
- **Rate limits**: Free tier has strict per-minute/per-day quotas. Handle 429 errors gracefully with fallback suggestions
- **Audio transcription**: Gemini supports audio via multimodal API. Convert PCM16 to WAV using `pcm16_to_wav()`, then use `protos.Part(inline_data=protos.Blob(mime_type="audio/wav", data=wav_bytes))`
- **Single API call optimization**: Combine transcription + suggestion in ONE API call to reduce latency. Use structured prompt with `TRANSCRIPT:`, `IS_QUESTION:`, `SUGGESTION:` format
- **SDK deprecation**: `google.generativeai` package is deprecated; future work should migrate to `google.genai`

### Configuration Pattern
- Backend config uses pydantic-settings (`backend/config.py`) with `.env` file
- Frontend config uses Next.js env vars (`NEXT_PUBLIC_*` prefix)
- Always provide `.env.example` templates when adding new env vars

### State Management Pattern
- Frontend uses Zustand store (`frontend/stores/sessionStore.ts`)
- Add new state fields to `SessionState` interface with corresponding setter actions
- Types defined in `frontend/types/index.ts`
- For data that should persist during session (like suggestions), use arrays instead of single values
- Only clear session data in the `reset()` action (called on session end)

### Auto-Scroll Pattern
- Use `useRef` with a bottom anchor element (`<div ref={bottomRef} />`) at end of scrollable content
- Use `scrollIntoView({ behavior: 'smooth', block: 'end' })` for smooth scrolling
- Wrap in `requestAnimationFrame()` to ensure DOM has updated before scrolling
- Trigger scroll in `useEffect` watching the data array (e.g., `[transcript]`, `[suggestions]`)

### Speech Completion Detection Pattern
- Use pause detection to determine when interviewer finishes speaking
- Accumulate transcript segments in a buffer during speech
- Monitor for silence (configurable threshold, default 2 seconds)
- Only generate suggestions after complete question is detected
- Run monitoring in a background asyncio task

### Cross-Window Communication Pattern
- Use BroadcastChannel API for same-origin cross-window communication
- Create channel with same name in both windows
- Broadcaster sends updates when state changes
- Receiver requests sync on load to get initial state
- Use `window.open()` with position/size params for popup windows

### Component Pattern
- Settings components go in `frontend/components/Settings/`
- Follow existing pattern: use `useSessionStore` hook, radio/checkbox inputs with styled labels
- Include helpful contextual info (badges, links) for user guidance

## 2025-01-27 - Bugfix: Audio Capture Not Working
- **What was implemented**: Fixed audio level meters not updating and system audio not being captured
- **Root causes**:
  1. AudioWorklet processor was reading from wrong input channels (`inputs[1]` instead of `inputs[0][1]`)
  2. ChannelMerger combines sources into channels of a single input, not separate inputs
- **Files changed**:
  - `frontend/public/audio-worklet-processor.js` - Fixed channel reading from `inputs[0][0]` (mic) and `inputs[0][1]` (system)
  - `frontend/hooks/useAudioCapture.ts` - Added debugging, better system audio detection, graceful fallback
- **Learnings for future iterations:**
  - ChannelMerger outputs multi-channel audio on a single input, not multiple inputs
  - In AudioWorklet: `inputs[0][0]` = first input's first channel, `inputs[0][1]` = first input's second channel
  - User MUST check "Share tab audio" / "Share audio" in browser screen share dialog for system audio to work
  - If no system audio track exists, create a silent oscillator to avoid breaking the audio pipeline
  - Always log audio track availability for debugging

## 2025-01-28 - Bugfix: Live Transcript and AI Suggestions Not Updating
- **What was implemented**: Fixed Gemini client to actually transcribe audio and generate suggestions instead of using mock data
- **Root causes**:
  1. `_process_audio()` was calling `_fallback_text_analysis()` which only generated random fake questions with 30% probability
  2. Audio data was being ignored entirely - no actual transcription was happening
  3. Two separate API calls (transcription + suggestion) caused unnecessary latency
- **Files changed**:
  - `backend/services/gemini_client.py`:
    - Added `pcm16_to_wav()` function to convert raw PCM16 audio to WAV format
    - Rewrote `_process_audio()` to use Gemini's multimodal API for real audio transcription
    - Combined transcription + suggestion into SINGLE API call (was 2 calls before)
    - Added `_parse_combined_response()` to extract transcript, is_question, and suggestion from structured response
    - Added comprehensive logging with `[AUDIO]`, `[PROCESS]`, `[PARSE]`, `[RECEIVE]` prefixes
  - `backend/routers/websocket.py`:
    - Added comprehensive logging with `[WS]`, `[WS-AUDIO]`, `[WS-RECV]`, `[SESSION]` prefixes
  - `backend/main.py`:
    - Added logging configuration to show all INFO+ messages with timestamps
- **Performance improvement**:
  - Before: ~9-13 seconds latency (5s audio + 2-4s transcription API + 2-4s suggestion API)
  - After: ~7-9 seconds latency (5s audio + 2-4s single combined API call)
- **Learnings for future iterations:**
  - Gemini DOES support audio via multimodal API using `protos.Part(inline_data=protos.Blob(mime_type="audio/wav", data=wav_bytes))`
  - Must convert PCM16 to WAV format before sending to Gemini (use `wave` module)
  - Combine multiple LLM operations into single API call when possible to reduce latency
  - Use structured response format (`TRANSCRIPT:`, `IS_QUESTION:`, `SUGGESTION:`) for reliable parsing
  - Add comprehensive logging at each step to debug audio pipeline issues
  - Audio buffer threshold of 30 chunks (~5 seconds) provides good balance of responsiveness vs. transcription quality

## 2025-01-29 - Feature: Wait for Complete Question Before Generating Suggestion
- **What was implemented**: AI now waits for interviewer to finish speaking before generating suggestions
- **Problem solved**: Previously suggestions were generated for partial sentences/fragments mid-speech
- **How it works**:
  1. Audio is transcribed immediately and shown in live transcript (for real-time feedback)
  2. Transcripts are accumulated in a buffer (`_transcript_buffer`)
  3. Background task monitors for speech pause (2 seconds of silence)
  4. When pause detected, complete accumulated transcript is analyzed
  5. If it's a question, suggestion is generated; otherwise ignored
- **Files changed**:
  - `backend/services/gemini_client.py`:
    - Added configuration constants:
      - `PAUSE_THRESHOLD_SECONDS = 2.0` - silence duration to consider speech complete
      - `MIN_QUESTION_LENGTH = 10` - minimum chars for valid question
    - Added new state variables:
      - `_transcript_buffer: list[str]` - accumulates transcript segments
      - `_last_speech_time: float` - timestamp of last detected speech
      - `_speech_completion_task` - background monitoring task
      - `_processing_suggestion` - prevents duplicate generation
    - Modified `_process_audio()`:
      - Now only does transcription (removed suggestion generation)
      - Sends transcripts immediately for live display
      - Accumulates transcripts and updates `_last_speech_time`
    - Added `_monitor_speech_completion()`:
      - Background asyncio task checking every 500ms
      - Detects pause when `time.time() - _last_speech_time >= PAUSE_THRESHOLD_SECONDS`
      - Triggers suggestion generation with complete transcript
    - Added `_generate_suggestion_for_complete_question()`:
      - Analyzes complete transcript to determine if it's a question
      - Uses structured prompt with `IS_QUESTION:` and `SUGGESTION:` format
      - Only sends suggestion to queue if it's actually a question
    - Updated `connect()` to start `_speech_completion_task`
    - Updated `disconnect()` to cancel task and clear buffers
- **Architecture decision**: Separated transcription from suggestion generation
  - Transcription: Immediate (for live display)
  - Suggestion: Delayed until speech complete (for accuracy)
- **Learnings for future iterations:**
  - Use background asyncio tasks for continuous monitoring (e.g., pause detection)
  - Separate concerns: live display vs. intelligent analysis can have different timing
  - Pause detection threshold of 2 seconds works well for natural speech patterns
  - Always clean up background tasks in `disconnect()` to prevent resource leaks
  - Use flags like `_processing_suggestion` to prevent duplicate/concurrent operations
  - The chat session (`self._chat`) provides context for better suggestion quality
