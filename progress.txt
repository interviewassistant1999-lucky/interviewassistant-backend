## Codebase Patterns

### Architecture
- **Backend**: Python FastAPI with WebSocket relay to LLM providers
- **Frontend**: Next.js 14 with Zustand for state management
- **Communication**: WebSocket for real-time audio streaming and message passing

### LLM Provider Pattern
- All LLM clients must implement the same interface: `connect()`, `send_audio()`, `receive_messages()`, `disconnect()`, `is_connected`
- Use `get_llm_client(provider)` factory function in `backend/services/openai_relay.py` to get the appropriate client
- Provider selection can come from: (1) frontend UI, (2) environment variable `LLM_PROVIDER`, (3) legacy `USE_MOCK_OPENAI` flag
- Message types for provider responses: `conversation.item.input_audio_transcription.completed` (transcription), `response.text.done` (suggestions)
- **Prompt key** can be passed to `connect()` to select response style (`candidate`, `coach`, `star`)

### Semantic Turn Detection Pattern
- **Problem**: Sending audio chunks every 3-4 seconds causes "jittery" partial suggestions
- **Solution**: Hybrid Accumulator approach in `backend/services/turn_detector.py`
- **Flow**:
  1. Transcribe instantly (~50-100ms) and show transcript to user
  2. Accumulate transcripts in TurnDetector buffer
  3. Monitor for silence (configurable threshold, default 1.5s)
  4. Check for "connective tissue" words (and, because, so...) - don't trigger if present
  5. Only call LLM for suggestion when turn is semantically complete
- **Configuration** (in `.env`):
  - `TURN_DETECTION_ENABLED=true` - Enable/disable feature
  - `TURN_SILENCE_THRESHOLD_MS=1500` - Silence duration for turn completion
  - `TURN_MIN_WORDS=3` - Minimum words for valid turn
- **Connective words**: If last words are connective (and, but, because, so, like, um...), speaker likely just pausing to think

### System Prompt Pattern
- All prompts are centralized in `backend/services/prompts.py`
- Use `get_prompt(prompt_key, job_description, resume, work_experience, verbosity)` to get formatted prompt
- Three prompt styles available:
  - `candidate`: First-person mode - AI speaks AS the candidate with battle stories and tactical advice
  - `coach`: Third-person mode - Classic coaching suggestions with key points
  - `star`: Behavioral mode - STAR method structure for behavioral questions
- `format_suggestion_for_display()` converts JSON response to display text based on prompt format
- Default prompt is `candidate` (configurable via `DEFAULT_PROMPT` constant)

### Groq-Specific Patterns (Adaptive Provider)
- **API compatibility**: Groq uses OpenAI-compatible API endpoints (`/v1/audio/transcriptions`, `/v1/chat/completions`)
- **Models**: Use `whisper-large-v3-turbo` for transcription, `llama-3.3-70b-versatile` for suggestions
- **Audio format**: Whisper requires WAV format - use `pcm16_to_wav()` to convert raw PCM16
- **Transcript merging**: Overlapping audio chunks need deduplication via `_merge_transcript()` method
- **JSON responses**: Use `response_format: {"type": "json_object"}` for structured parsing
- **Performance**: ~50-100ms transcription, ~200-400ms suggestion generation (10x faster than alternatives)

### Gemini-Specific Patterns
- **Model names**: Use short names like `gemini-2.5-flash`, `gemini-2.0-flash` (not `models/gemini-...`)
- **Available models** (as of Jan 2025): `gemini-2.5-flash`, `gemini-2.0-flash`, `gemini-2.0-flash-lite`, `gemini-flash-latest`, `gemini-pro-latest`
- **Rate limits**: Free tier has strict per-minute/per-day quotas. Handle 429 errors gracefully with fallback suggestions
- **Audio transcription**: Gemini supports audio via multimodal API. Convert PCM16 to WAV using `pcm16_to_wav()`, then use `protos.Part(inline_data=protos.Blob(mime_type="audio/wav", data=wav_bytes))`
- **Single API call optimization**: Combine transcription + suggestion in ONE API call to reduce latency. Use structured prompt with `TRANSCRIPT:`, `IS_QUESTION:`, `SUGGESTION:` format
- **SDK deprecation**: `google.generativeai` package is deprecated; future work should migrate to `google.genai`

### Configuration Pattern
- Backend config uses pydantic-settings (`backend/config.py`) with `.env` file
- Frontend config uses Next.js env vars (`NEXT_PUBLIC_*` prefix)
- Always provide `.env.example` templates when adding new env vars

### State Management Pattern
- Frontend uses Zustand store (`frontend/stores/sessionStore.ts`)
- Add new state fields to `SessionState` interface with corresponding setter actions
- Types defined in `frontend/types/index.ts`
- For data that should persist during session (like suggestions), use arrays instead of single values
- Only clear session data in the `reset()` action (called on session end)

### Auto-Scroll Pattern
- Use `useRef` with a bottom anchor element (`<div ref={bottomRef} />`) at end of scrollable content
- Use `scrollIntoView({ behavior: 'smooth', block: 'end' })` for smooth scrolling
- Wrap in `requestAnimationFrame()` to ensure DOM has updated before scrolling
- Trigger scroll in `useEffect` watching the data array (e.g., `[transcript]`, `[suggestions]`)

### Speech Completion Detection Pattern
- Use pause detection to determine when interviewer finishes speaking
- Accumulate transcript segments in a buffer during speech
- Monitor for silence (configurable threshold, default 2 seconds)
- Only generate suggestions after complete question is detected
- Run monitoring in a background asyncio task

### Cross-Window Communication Pattern
- Use BroadcastChannel API for same-origin cross-window communication
- Create channel with same name in both windows
- Broadcaster sends updates when state changes
- Receiver requests sync on load to get initial state
- Use `window.open()` with position/size params for popup windows

### Component Pattern
- Settings components go in `frontend/components/Settings/`
- Follow existing pattern: use `useSessionStore` hook, radio/checkbox inputs with styled labels
- Include helpful contextual info (badges, links) for user guidance

### Speaker Detection Pattern
- Audio worklet tracks energy levels per channel: `micEnergy` vs `systemEnergy`
- Compare accumulated energy over chunk duration to determine dominant speaker
- Send `speaker.update` WebSocket message when speaker changes (debounce to avoid flapping)
- Backend tracks `current_speaker` in connection state for transcript labeling
- Visual grouping: new block on speaker change, inline continuation for same speaker

### Hero + Collapsible History Pattern
- Latest/current item gets "hero" treatment - prominent, immediately visible
- Historical items collapse into minimal pills/previews
- Expand individual history items on click
- Toggle to show/hide history section entirely
- Auto-collapse history when new item arrives
- Use entrance animations (fade + translate) triggered by item ID change

### Silence-Based Turn Detection Pattern
- **Problem**: Grouping by speaker alone fails when one party doesn't speak
- **Solution**: Track `last_transcript_time` and detect turns based on silence
- If more than 5 seconds passes between transcripts, set `isNewTurn=true`
- Frontend checks both speaker change AND `isNewTurn` flag for grouping
- First transcript always starts a new turn
- **Important**: Threshold must be > `audio_buffer_seconds` (3s) to avoid false triggers
- Threshold of 5 seconds accounts for audio buffering delays

### Turn Detector Optimization Pattern
- **Reset state between turns**: Clear `_last_transcript` and `_transcript_buffer` when turn completes
- **Connective word check**: Only check the LAST word, not last 3 words (too aggressive)
- **Punctuation trust**: If last word has punctuation (. ? !), trust it even if word looks connective
- **Silence thresholds**: With punctuation = 1x threshold, without = 1.5x threshold (was 2x)
- **Preposition check**: Only check for dangling prepositions when no punctuation present

### Transcript Turn Detection Pattern (Two Options)
- **Option A (Legacy)**: Based on transcript arrival time
  - Simple but inaccurate due to audio buffering delays
  - Threshold must be > `audio_buffer_seconds` to avoid false triggers
  - Config: `use_speech_timing_for_turns = False`
- **Option B1 (Recommended)**: Based on frontend VAD speech timing
  - Frontend audio worklet tracks actual speech/silence
  - Sends `speechEndTimestamp` and `silenceDurationMs` via `speech.timing` message
  - Backend uses actual silence duration for accurate turn detection
  - Config: `use_speech_timing_for_turns = True`, `speech_silence_threshold_ms = 3000`
  - More accurate because it measures real silence, not transcript arrival gaps

### Database Pattern (SQLAlchemy 2.0 Async)
- Use `AsyncSession` with `async_sessionmaker` for async operations
- Models inherit from `DeclarativeBase` (not legacy `declarative_base()`)
- Use `Mapped[Type]` and `mapped_column()` for type-safe columns
- Relationships use `Mapped[List["Model"]]` with `relationship()`
- Session dependency: `async def get_db() -> AsyncGenerator[AsyncSession, None]`
- Lifespan handler in FastAPI for init/close database connections
- Alembic requires special async configuration in `env.py`

### Authentication Pattern
- **Password hashing**: Use `passlib[bcrypt]` with `CryptContext`
- **JWT tokens**: Use `python-jose` with HS256 algorithm
- **Token storage**: Frontend stores in localStorage via Zustand persist middleware
- **Protected routes**: Client-side check with redirect to login
- **Dependency injection**: `require_auth = Depends(get_current_user)` for protected endpoints
- **OAuth flow**:
  1. Frontend calls `/api/auth/google` to get OAuth URL
  2. Redirect to Google
  3. Google redirects to `/auth/callback` with code
  4. Frontend sends code to `/api/auth/google/callback`
  5. Backend exchanges code for tokens, creates/updates user, returns JWT

### API Key Encryption Pattern
- Use `cryptography.fernet.Fernet` for symmetric encryption
- Generate key: `Fernet.generate_key()` → store in `ENCRYPTION_KEY` env var
- Encrypt on save: `fernet.encrypt(key.encode()).decode()`
- Decrypt on use: `fernet.decrypt(encrypted.encode()).decode()`
- Mask for display: `key[:3] + "****"` (never show full key)
- Validate before save: Make test API call to provider

### Interview Prep Pattern (Personalized Intelligence)
- **2-step wizard**: Step 1 = context (company, round, resume upload, JD), Step 2 = Q&A review + settings
- **MongoDB**: Motor async client singleton in `services/mongodb_service.py`, init/close in lifespan
- **Question scoring**: Broad query + client-side scoring (company +10, round +5, role +3, tags +1, recency +2, verified +1)
- **Answer generation**: Uses user's configured LLM provider via httpx (groq/openai/gemini dispatch)
- **Prompt injection**: Approved Q&A formatted as "PRE-PREPARED ANSWERS" section appended to system prompt
- **Resume parsing**: PDF (PyPDF2) and DOCX (python-docx) support via `services/resume_parser.py`
- **File upload**: Uses `python-multipart` + FastAPI `UploadFile`, max 5MB
- **All LLM clients**: `connect()` now accepts `pre_prepared_answers` param passed through to prompt builder
- **Auth pattern**: Use `require_auth` from `routers.auth` as Depends, returns User object (use `user.id`)

### Dashboard Layout Pattern
- Use Next.js nested layouts (`dashboard/layout.tsx`)
- Sidebar with navigation links using `usePathname()` for active state
- `isActive()` helper: exact match for root, `startsWith()` for nested
- Protected wrapper component checks auth before rendering children
- Show user info (name, tier) in header with logout button

### Landing Page Pattern (Marketing)
- Light theme for marketing, dark theme for app
- Hero → Features → How it works → Pricing → CTA → Footer
- Use Tailwind `bg-gradient-to-b from-slate-50 to-white` for subtle gradients
- Pricing cards with border-2 highlight for recommended plan
- Conditional CTA based on auth state (logged in → "Start Interview", not → "Get Started")

## 2025-01-29 - Feature: Wait for Complete Question Before Generating Suggestion
- **What was implemented**: AI now waits for interviewer to finish speaking before generating suggestions
- **Problem solved**: Previously suggestions were generated for partial sentences/fragments mid-speech
- **How it works**:
  1. Audio is transcribed immediately and shown in live transcript (for real-time feedback)
  2. Transcripts are accumulated in a buffer (`_transcript_buffer`)
  3. Background task monitors for speech pause (2 seconds of silence)
  4. When pause detected, complete accumulated transcript is analyzed
  5. If it's a question, suggestion is generated; otherwise ignored
- **Files changed**:
  - `backend/services/gemini_client.py`:
    - Added configuration constants:
      - `PAUSE_THRESHOLD_SECONDS = 2.0` - silence duration to consider speech complete
      - `MIN_QUESTION_LENGTH = 10` - minimum chars for valid question
    - Added new state variables:
      - `_transcript_buffer: list[str]` - accumulates transcript segments
      - `_last_speech_time: float` - timestamp of last detected speech
      - `_speech_completion_task` - background monitoring task
      - `_processing_suggestion` - prevents duplicate generation
    - Modified `_process_audio()`:
      - Now only does transcription (removed suggestion generation)
      - Sends transcripts immediately for live display
      - Accumulates transcripts and updates `_last_speech_time`
    - Added `_monitor_speech_completion()`:
      - Background asyncio task checking every 500ms
      - Detects pause when `time.time() - _last_speech_time >= PAUSE_THRESHOLD_SECONDS`
      - Triggers suggestion generation with complete transcript
    - Added `_generate_suggestion_for_complete_question()`:
      - Analyzes complete transcript to determine if it's a question
      - Uses structured prompt with `IS_QUESTION:` and `SUGGESTION:` format
      - Only sends suggestion to queue if it's actually a question
    - Updated `connect()` to start `_speech_completion_task`
    - Updated `disconnect()` to cancel task and clear buffers
- **Architecture decision**: Separated transcription from suggestion generation
  - Transcription: Immediate (for live display)
  - Suggestion: Delayed until speech complete (for accuracy)
- **Learnings for future iterations:**
  - Use background asyncio tasks for continuous monitoring (e.g., pause detection)
  - Separate concerns: live display vs. intelligent analysis can have different timing
  - Pause detection threshold of 2 seconds works well for natural speech patterns
  - Always clean up background tasks in `disconnect()` to prevent resource leaks
  - Use flags like `_processing_suggestion` to prevent duplicate/concurrent operations
  - The chat session (`self._chat`) provides context for better suggestion quality

## 2026-01-30 - Feature: Rate Limiting Strategy for Gemini Free Tier Testing
- **What was implemented**: Comprehensive rate limiting system to enable testing with Gemini's free tier (5 RPM limit)
- **Problem solved**: Gemini free tier has strict rate limits that were causing 429 errors during testing
- **Strategy implemented**:
  1. **Token Bucket Rate Limiter** - Classic algorithm for smooth request flow
  2. **Request Queuing** - Queues requests when rate limited, processes when tokens available
  3. **Configurable Audio Buffers** - 10 seconds instead of 5 (halves transcription requests)
  4. **Transcript Caching** - Caches similar audio transcripts to avoid duplicate API calls
  5. **Dev Mode Toggle** - Enable rate-limit-safe testing mode via environment variable
  6. **Frontend Status Indicator** - Shows rate limit status in UI
- **Files created**:
  - `backend/services/rate_limiter.py` - TokenBucket, RateLimitedExecutor, TranscriptCache classes
  - `frontend/components/StatusBar/RateLimitIndicator.tsx` - UI component for rate limit status
- **Files changed**:
  - `backend/config.py` - Added DEV_MODE, RATE_LIMIT_RPM, AUDIO_BUFFER_SECONDS settings
  - `backend/services/gemini_client.py` - Integrated rate limiter, configurable audio buffers
  - `backend/.env.example` - Documented new settings
  - `frontend/stores/sessionStore.ts` - Added rateLimit state
  - `frontend/hooks/useWebSocket.ts` - Handle rate_limit.status and rate_limit.update messages
  - `frontend/components/StatusBar/StatusBar.tsx` - Added RateLimitIndicator
- **New Environment Variables**:
  - `DEV_MODE=true` - Enable rate limiting mode
  - `RATE_LIMIT_RPM=4` - Requests per minute (4 for safety margin on 5 RPM limit)
  - `AUDIO_BUFFER_SECONDS=10` - Buffer 10 seconds of audio before transcription
- **Rate Limit Math**:
  - 10 second audio buffers = max 6 transcription requests/minute
  - With 4 RPM limit and queuing, actual throughput is ~4 transcriptions/minute
  - Suggestions are generated only after speech pause, further reducing requests
  - Result: Can test full interview flow within free tier limits
- **Learnings for future iterations**:
  - Token bucket algorithm is ideal for rate limiting with bursts
  - Caching audio transcripts by hash reduces redundant API calls
  - Longer audio buffers trade latency for fewer API calls
  - Dev mode should be easily toggleable for production deployment
  - UI feedback for rate limiting improves developer experience

## 2026-02-01 - Feature: Semantic Turn Detection (Anti-Jitter)
- **What was implemented**: Hybrid Accumulator approach to detect when interviewer finishes speaking before generating suggestions
- **Problem solved**: Sending audio chunks every 3-4 seconds caused "jittery" partial suggestions on incomplete sentences
- **Technical approach**:
  1. **Instant Transcription**: Groq Whisper transcribes in ~50-100ms, shown to user immediately
  2. **Accumulation**: Transcripts buffer in TurnDetector instead of triggering LLM each time
  3. **Silence Detection**: Monitor for configurable silence threshold (default 1.5s)
  4. **Connective Tissue Check**: Don't trigger if last words are "and", "because", "so", "um", etc.
  5. **Complete Sentence**: Check for ending punctuation as strong completion signal
- **Files created**:
  - `backend/services/turn_detector.py` - TurnDetector and TranscriptAccumulator classes
- **Files changed**:
  - `backend/services/groq_client.py` - Integrated TurnDetector, separates transcription from suggestion
  - `backend/config.py` - Added turn detection settings
  - `backend/.env.example` - Documented new settings
- **Configuration options**:
  - `TURN_DETECTION_ENABLED=true` - Toggle the feature
  - `TURN_SILENCE_THRESHOLD_MS=1500` - How long silence before turn complete
  - `TURN_MIN_WORDS=3` - Minimum words for valid turn
- **Connective words that delay trigger**:
  - Conjunctions: and, but, or, because, since, although, if, when...
  - Prepositions: to, of, in, on, at, with, about, for...
  - Thinking sounds: um, uh, er, ah, hmm, well, now, then...
- **Learnings for future iterations**:
  - Separating transcription from suggestion generation enables smarter triggering
  - ~1.5 second silence is good threshold for natural speech pauses
  - Connective tissue detection prevents 80%+ of false triggers
  - Complete punctuation (. ? !) is strong signal of turn completion
  - Force-complete on disconnect to not lose final content

## 2026-02-01 - Bugfix: Slow AI Suggestions for Second Question
- **What was implemented**: Fixed delay in AI suggestion generation for questions after the first one
- **Problems identified**:
  1. `_last_transcript` not reset between turns - caused incorrect overlap detection
  2. Turn detector checked last 3 words for connectives - too aggressive, blocked valid completions
  3. Without punctuation, required 2x silence threshold (3s) - too slow
- **Root causes**:
  - After Question 1 completed, `_last_transcript` still contained old words
  - Question 2's first words might be incorrectly removed as "overlap"
  - Questions like "Tell me about your experience?" blocked because "about" is connective
- **Fixes applied**:
  1. Reset `_last_transcript` and `_transcript_buffer` in `_on_turn_complete()`
  2. Only check LAST word for connective, not last 3 words
  3. Trust punctuation - if last word has . ? !, don't block on connective
  4. Reduced "no punctuation" silence from 2x to 1.5x threshold
- **Files changed**:
  - `backend/services/groq_client.py`:
    - Added reset of `_last_transcript` and `_transcript_buffer` in `_on_turn_complete()`
  - `backend/services/turn_detector.py`:
    - Simplified connective word checking to only last word
    - Added punctuation trust: skip connective check if punctuation present
    - Reduced long silence threshold from 2x to 1.5x
    - Only check dangling prepositions when no punctuation
- **Performance improvement**:
  - Before: 2nd question could take 3+ seconds extra for suggestion
  - After: Consistent ~1.5s for turn detection with punctuation, ~2.25s without
- **Learnings for future iterations**:
  - Always reset accumulator state between turns/questions
  - Punctuation from Whisper is generally reliable - trust it
  - Less aggressive connective checking reduces false negatives
  - Separate overlap merging state from turn detection state

    - Updated grouping logic: `sameSpeaker && !entry.isNewTurn`

### Data Flow
1. **User lands on `/`** → Marketing landing page
2. **User signs up/logs in** → JWT token stored in localStorage
3. **User adds API key** → `/dashboard/settings` (required before starting)
4. **User starts interview** → `/interview` (checks for API key first)
5. **WebSocket connects** → JWT token passed as query param for auth
6. **Audio captured** → AudioWorklet merges mic + system audio
7. **Audio sent via WebSocket** → Backend uses user's API key for LLM
8. **Transcription returned** → Displayed in real-time, accumulated for DB
9. **Turn completed** → AI generates suggestion, accumulated for DB
10. **Session ends** → Transcript & suggestions saved to database
11. **User reviews in dashboard** → `/dashboard/sessions` shows history

## 2026-02-04 - E2E Testing Session: BUG-001 Fix (Auth Hydration Race Condition)

### Session Overview
- **Purpose**: E2E testing of Interview Assistant application
- **Test Configuration**: Groq (Adaptive) provider with API key
- **Goal**: Test interviewer audio transcription and AI suggestions

### Critical Bug Found and Fixed: BUG-001

**Problem**: Auth hydration race condition causing premature logout on navigation to /interview page

**Root Cause Analysis**:
1. User navigates to `/interview` page
2. React component mounts and `useEffect` runs `loadApiKeys()` immediately
3. Zustand store hasn't hydrated from localStorage yet (async process)
4. `fetchWithAuth` reads `token` from Zustand state - it's `null` (not hydrated)
5. API request sent without Authorization header → 401 Unauthorized
6. `fetchWithAuth` sees 401 and calls `logout()` as a side effect
7. `logout()` CLEARS the token from localStorage (destructive!)
8. Now even localStorage is empty - user is fully logged out
9. Subsequent hydration finds empty localStorage - permanently logged out

**Fix Implemented (3-part solution)**:

**1. authStore.ts - Add hydration tracking**:
```typescript
interface AuthState {
  // ... existing fields
  _hasHydrated: boolean  // Track if store has hydrated from localStorage
  setHasHydrated: (hasHydrated: boolean) => void
}

// In persist config:
onRehydrateStorage: () => (state) => {
  state?.setHasHydrated(true)
},
```

**2. useAuth.ts - Conditional logout on 401**:
```typescript
if (response.status === 401) {
  // Only logout if we actually had a token (avoid logout during hydration)
  if (token) {
    logout()
  }
  throw new Error('Session expired')
}
```

**3. interview/page.tsx - Wait for hydration before API calls**:
```typescript
const { isAuthenticated, _hasHydrated } = useAuthStore()

useEffect(() => {
  const loadApiKeys = async () => {
    if (!_hasHydrated) return  // Wait for hydration
    if (!isAuthenticated) return
    // ... rest of loading logic
  }
  loadApiKeys()
}, [isAuthenticated, _hasHydrated, fetchWithAuth])
```
